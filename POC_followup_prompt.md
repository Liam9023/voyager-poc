## Completed & tested

1. **Itinerary editing — Add, Remove, Swap.** Shared conflict-checking mechanism (`lib/schedule.ts`, `ConflictSheet.tsx`) surfaces overlaps/clashes and requires confirm before adjusting anything else. "Tell Me More" opens Ask pre-loaded with activity context (`ActivityRow.tsx`).
2. **Pacing & Decision Style onboarding preferences.** Both selectors live in conversation mode (inferred via `lib/infer-preferences.ts`) and as explicit quick-selects (`app/plan/page.tsx`). Pacing caps activities/day in `DayCard.tsx`; Decision Style drives alternative counts in `lib/alternatives.ts`.
3. **Settings → Preferences section.** Home airport, airlines, dietary requirements, food dislikes, accessibility needs, general notes — persisted via `lib/preferences-store.tsx` (localStorage), editable from Settings, confirmed to survive a page refresh, and folded into Ask context via `lib/prompts.ts`.
4. **Manual booking entry.** `AddManualBooking` on the Bookings tab (`app/trip/page.tsx`) saves type/title/detail/date/optional reference into shared bookings state.
5. **Google Places API integration for venue grounding**, including the priority-order booking fallback and rating/price/Maps-link social proof. `lib/places-service.ts` field mask now carries name, rating, userRatingCount, priceLevel, editorialSummary, websiteUri, googleMapsUri, nationalPhoneNumber. The "See recent posts" Instagram button has been removed entirely (`lib/social-links.ts` deleted). `components/VenueBookAction.tsx` resolves a real per-venue booking action — website → Google Maps listing → phone ("Call to book") → no button at all — for any category without a direct provider deep link (dining, car hire, rail, transfers), replacing the old plain-Google-search fallback. `components/VenueBadge.tsx` shows a star rating + review count and price level badge plus a "See on Google Maps" link on every venue card in the itinerary and swap/add alternatives. `lib/prompts.ts` now instructs Claude to weigh confidence by rating AND review count together (tentative language for high-rating/low-review-count venues).
6. **Conversational tone rewrite.** `TONE_RULES` in `lib/prompts.ts` encodes lead-with-answer, sentences-over-bullets, stated preference, matched length, no context re-explaining, and concrete-detail-over-adjectives — applied across conversation, Ask, and itinerary-facing prompts.
7. **Real deep links for booking actions.** `lib/deep-links.ts` builds real Booking.com, Google Flights (labeled "Flights", no provider name shown), and Viator deep links with real params; `BookLink.tsx` opens all in a new tab. Categories without genuine provider coverage (car hire, dining, rail, transfers) route through the live-resolved `VenueBookAction` from item 5 instead of a canned search URL.
8. **"Ask about this day" entry point, fixed.** A visible chip on every day card opens Ask scoped to that day (`app/trip/page.tsx`). The day's detail is injected as invisible background context (`elementContext`, unchanged) — Voyager now opens with a short natural line ("What's on your mind for Noosa National Park?") instead of auto-firing a fake user question, and the persistent header (`AskPanel.tsx`) dynamically shows the actual day/activity (e.g. "Day 8 · The Cotswolds") instead of a hardcoded "Day 2 · Noosa Heads".
9. **Hero photo on each day card via Unsplash.** `lib/unsplash-service.ts` + `/api/day-photo` (and `/api/day-photo/track` for the required download-tracking ping) search Unsplash per day location, hotlinked directly with photographer/Unsplash attribution (`components/DayHeroPhoto.tsx`), cached client- and server-side per query. Verified live against the real Unsplash API.
10. **Crowd-avoidance logic using review count.** `lib/prompts.ts` instructs Claude to treat very-high-review-count venues as a busy/tourist-trap signal and steer toward well-rated, less-saturated alternatives when the traveller mentions avoiding crowds — explaining the reasoning briefly. Verified live against the real Claude API: correctly compared review counts (e.g. "over 1,000 reviews" vs. "388 reviews") to steer recommendations.
11. **Group expense tracking.** New "💰 Expenses" 5th sub-tab in the Trip tab (`app/trip/page.tsx`), alongside Alerts/Packing/Tools/Settings. Each expense has title/amount/payer/date/optional category and a per-expense split (`lib/travellers.ts` — Liam & Sarah); `lib/expenses.ts` computes each traveller's paid/owed/net balance. Persisted via the same localStorage pattern as the rest of the trip state (`lib/store.tsx`). Verified in a real browser: a $200 expense paid by Liam and split evenly correctly shows Sarah owing $100, persists across a page reload, with no console errors.
